# Template Generator

## Overview

Use this skill to convert stable project memory into a reusable project template.

This skill does **not** scan raw source code directly unless needed for clarification. It primarily consumes `.docs/` generated by `project-scanner`.

**Announce at start:**

> spark template 🧩 Using template-generator to build reusable project blueprint

## Project Memory Input

Read `.docs/` first.

If the workspace contains multiple `.docs/` directories, identify which valid project root should be templated and use that project's memory set only. Do not merge multiple project memories into one template unless the developer explicitly asks for a composed workspace template.

Expected project memory set:

```txt
.docs/
├── PROJECT_SCAN.md
├── PROJECT_PROFILE.md
├── ARCHITECTURE_GRAPH.md
├── WORKSPACE_MAP.md
├── FEATURE_MAP.md
├── API_CONTRACT.md
├── DATABASE_CATALOG.md
├── BUSINESS_FLOW.md
├── DOMAIN_MAP.md
└── TESTING_STRATEGY.md
```

## Output

```txt
.template/
├── TEMPLATE_PROFILE.md
├── TEMPLATE_WORKFLOW.md
├── FOLDER_STRUCTURE.md
├── FILE_STRUCTURE.md
├── CODE_STRUCTURE.md
├── ROUTING_STRUCTURE.md
├── API_STRUCTURE.md
├── DATABASE_STRUCTURE.md
├── TESTING_STRUCTURE.md
└── GENERATION_RULES.md
```

You must physically create `.template/` and write the files to disk.

## Responsibility

This skill is a **Template Blueprint Builder**.

It converts:

```txt
Project Memory
↓
Reusable Template Blueprint
```

It must normalize project-specific knowledge into reusable conventions.

## Rules

* Read `.docs/` first.
* Do not generate a template if `.docs/` is missing or incomplete.
* Do not copy project-specific business data directly into the template.
* Convert concrete names into reusable placeholders.
* Preserve proven architecture, structure, routing, API, database, and testing conventions.
* Mark uncertain conventions as `Needs Review`.
* Do not generate a new project. That is the responsibility of `project-onboarding`.

## Normalization Rules

Convert project-specific facts into template patterns.

Examples:

```txt
EmployeeProfile.vue -> <FeatureName>Profile.vue
employee.service.ts -> <feature>.service.ts
GET /employee -> GET /<resource>
employees table -> <resource_plural> table
EmployeeModule -> <FeatureName>Module
```

## Required Template Files

### TEMPLATE_PROFILE.md

Must describe:

* template name
* template purpose
* source project memory
* supported project type
* recommended tech stack
* package manager
* runtime requirements
* intended use cases
* unsupported use cases

### TEMPLATE_WORKFLOW.md

Must be the most detailed file.

It must explain:

* project generation flow
* folder creation flow
* file creation flow
* routing flow
* API flow
* database/model flow
* validation flow
* testing flow
* how to add a new feature
* how to add a new route
* how to add a new API client or endpoint
* how to add a new module/component
* how to extend the template safely

### FOLDER_STRUCTURE.md

Must document the recommended folder layout.

Include:

* root-level folders
* source folders
* feature/module folders
* shared/common folders
* config folders
* test folders
* generated folders
* ignored folders

### FILE_STRUCTURE.md

Must document standard files and naming conventions.

Include:

* required files
* optional files
* file naming rules
* feature file patterns
* config file patterns
* test file patterns

### CODE_STRUCTURE.md

Must document code organization rules.

Include:

* layering convention
* component/module structure
* business logic placement
* validation placement
* type/interface placement
* error handling pattern
* reusable utility pattern

### ROUTING_STRUCTURE.md

Must document routing conventions.

Include:

* route file location
* route naming
* path naming
* lazy loading convention
* nested routing convention
* route guard convention if proven
* menu/navigation relationship if proven

### API_STRUCTURE.md

Must document API conventions.

Include:

* API client or endpoint structure
* request/response model pattern
* DTO/schema convention
* error response handling
* authentication header pattern if proven
* environment base URL convention if proven

### DATABASE_STRUCTURE.md

Must document database conventions when applicable.

Include:

* entity/model pattern
* schema location
* migration convention
* repository/data-access pattern
* naming convention
* relationship convention

If the source project has no database layer, mark this as `Not Applicable`.

### TESTING_STRUCTURE.md

Must document testing conventions.

Include:

* test framework
* test folder layout
* unit test pattern
* integration test pattern
* e2e test pattern
* test naming convention
* test command surface

### GENERATION_RULES.md

Must define how `project-onboarding` should use this template.

Include:

* required input variables
* optional input variables
* placeholder replacement rules
* files to rename
* files to update
* commands to run after generation
* validation checks
* failure handling

## Required Metadata

Every template file must include:

* Generated At
* Source Memory
* Confidence
* Evidence
* Needs Review
