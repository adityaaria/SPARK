# Template Generator

## Overview

Use this skill to convert stable project memory into a reusable project template blueprint.

This skill does **not** scan raw source code directly unless clarification is truly required. It consumes `.docs/` generated by `project-scanner`, treats Stable Memory as the only source of truth, and projects a reusable `.template/` from an internal engineering blueprint.

**Announce at start:**

> spark template 🧩 Using template-generator to build reusable project blueprint

---

# Project Memory Input

Read `.docs/` first.

If the workspace contains multiple `.docs/` directories, identify which valid project root should be templated and use that project's memory set only. Do not merge multiple project memories into one template unless the developer explicitly asks for a composed workspace template.

Expected project memory set:

```txt
.docs/
├── PROJECT_SCAN.md
├── PROJECT_PROFILE.md
├── ARCHITECTURE_GRAPH.md
├── WORKSPACE_MAP.md
├── FEATURE_MAP.md
├── API_CONTRACT.md
├── DATABASE_CATALOG.md
├── BUSINESS_FLOW.md
├── DOMAIN_MAP.md
└── TESTING_STRATEGY.md
```

Stable Memory is the only source of truth.

Do **not** scan the repository unless clarification is truly required because the memory is incomplete or contradictory.

---

# Output Contract

Generate exactly these files and no others:

```txt
.template/
├── TEMPLATE_PROFILE.md
├── TEMPLATE_WORKFLOW.md
├── FOLDER_STRUCTURE.md
├── FILE_STRUCTURE.md
├── CODE_STRUCTURE.md
├── ROUTING_STRUCTURE.md
├── API_STRUCTURE.md
├── DATABASE_STRUCTURE.md
├── TESTING_STRUCTURE.md
└── GENERATION_RULES.md
```

You must physically create `.template/` and write the files to disk.

Do not create new permanent files.

---

# Responsibility

This skill is a **Template Blueprint Builder**.

It converts:

```txt
Stable Project Memory
↓
Engineering Pattern Extraction
↓
Architecture Abstraction
↓
Engineering Blueprint
↓
Reusable Template Projection
↓
.template
```

The generated template must be an engineering blueprint, not a generalized copy of the source project.

The **Engineering Blueprint** exists only during execution. It is internal, temporary, and must never be persisted.

---

# Template Thinking Rules

Before generating templates, apply these rules:

- prefer engineering pattern over project structure
- prefer convention over implementation
- prefer architecture over technology
- prefer generic contract over concrete naming
- prefer reusable pattern over business logic
- prefer architecture decisions over folder names
- extract why before how
- extract reusable engineering knowledge before implementation details
- evidence overrides assumptions
- unknown is better than incorrect
- never generalize unsupported patterns

These rules are mandatory.

---

# Rules

- Read `.docs/` first.
- Do not generate a template if `.docs/` is missing or incomplete.
- Do not scan raw source code unless clarification is truly required.
- Do not generate implementation code.
- Do not generate a new project. That is the responsibility of `project-onboarding`.
- Do not copy project-specific business data directly into the template.
- Convert concrete names into reusable placeholders.
- Preserve proven architecture, structure, routing, API, database, testing, and UI architecture conventions.
- Mark uncertain conclusions as `Needs Review`.
- Keep output deterministic.
- Keep output polyglot and framework agnostic.

# Pattern Discovery

Extract reusable engineering patterns from Stable Memory only.

Look for:

- architecture layers
- workspace conventions
- module boundaries
- folder conventions
- file naming patterns
- routing conventions
- API contract patterns
- persistence patterns
- testing strategy
- validation placement
- configuration strategy
- integration boundaries
- design-system structure
- reusable component hierarchy
- extension points
- anti-patterns
- evolution rules

Prefer patterns that appear as stable conventions across the memory set, not isolated implementation details.

---

# Pattern Validation

Validate every candidate pattern before it becomes reusable template knowledge.

Check:

- is it supported by Stable Memory evidence
- is it repeated enough to be treated as a convention
- is it architecture-specific, framework-specific, optional, or replaceable
- is it business-specific and therefore non-reusable
- is it implementation leakage rather than reusable engineering knowledge

If evidence is insufficient, mark `Needs Review` instead of inventing a rule.

---

# Pattern Generalization

Convert project-specific facts into reusable template patterns.

Examples:

```txt
EmployeeProfile.vue -> <FeatureName>Profile.<view_ext>
employee.service.ts -> <feature>.service.<lang_ext>
GET /employee -> GET /<resource>
employees table -> <resource_plural> table
EmployeeModule -> <FeatureName>Module
```

Generalize names, not architecture. Preserve the reusable pattern while removing business specificity.

Never carry over:

- project-specific feature names
- project-specific endpoints
- organization-specific rules
- business-specific styling
- business-specific workflows presented as universal patterns

---

# Architecture Abstraction

Abstract into architecture before mentioning frameworks or libraries.

Prefer architecture such as:

- presentation layer
- application layer
- state layer
- integration layer
- infrastructure layer

Framework implementations are supporting evidence only.

Implementation is always the last layer.

Architecture is always the first layer.

---

# Engineering Decision Extraction

For every discovered convention, determine when possible:

- why it exists
- what problem it solves
- whether it is mandatory
- whether it is optional
- whether it is replaceable
- whether it is framework-specific
- whether it is architecture-specific

When evidence is strong enough, express decisions as engineering contracts using:

- `MUST`
- `SHOULD`
- `MAY`
- `MUST NOT`

Examples:

- Views MUST NOT access transport clients directly.
- Presentation layer SHOULD remain business-agnostic.
- Business logic MUST stay outside reusable UI components.
- Services MUST own orchestration.
- Transport layer MUST remain infrastructure-only.
- DTOs MUST remain transport contracts.
- Shared libraries MUST avoid business-specific behavior.

If evidence is insufficient, mark `Needs Review`. Never invent decisions.

---

# Variability Analysis

Classify each reusable pattern only when supported by evidence:

- Mandatory
- Optional
- Replaceable
- Framework-specific
- Architecture-specific
- Project-specific
- Business-specific

Examples of things that may need classification:

- authentication
- UI library
- state management
- module federation
- transport layer
- styling strategy
- testing framework
- build tool

This classification exists to improve template adaptability without collapsing architecture into implementation.

# Engineering Blueprint

The internal Engineering Blueprint is the shared reasoning layer for template generation.

It must capture reusable:

- engineering patterns
- architecture abstractions
- engineering decisions
- variability classifications
- extension rules
- evolution rules
- anti-patterns
- UI architecture patterns

It is temporary and must never be written as its own file.

# Template Projection

The ten `.template` files are projections of the internal Engineering Blueprint.

This means:

- each file reads from the same internal reasoning layer
- no file independently invents rules
- terminology remains consistent across the full template set
- implementation details appear only as supporting evidence or replaceable examples
- output quality increases through reasoning quality, not document length

Do not enlarge documents merely to add sections. Increase knowledge density instead.

---

# Design System Extraction

Extract reusable design-system architecture. Never simply copy UI implementation.

Extract when supported by Stable Memory:

- Design System Strategy
- Component Hierarchy
- Component Ownership
- Design Tokens
- Spacing Strategy
- Typography Strategy
- Theme Strategy
- Layout Strategy
- Responsive Strategy
- Accessibility Strategy
- Interaction Pattern
- Visual Composition Pattern

Design system knowledge must remain business-neutral.

Business-specific styling must never become reusable design-system knowledge.

Treat design system as reusable UI architecture, not as a Vue/Tailwind/library-specific implementation.

---

# Reusable Component Extraction

Extract reusable component architecture such as:

- Primitive Components
- Composed Components
- Feature Components
- Page Templates
- Application Views

Document ownership and responsibility for each layer when supported by evidence.

Avoid implementation-specific assumptions.

---

# Styling Strategy Abstraction

Do not assume a specific styling tool or UI library.

Extract only when supported by evidence:

- Styling Strategy
- Theme Strategy
- Token Strategy
- Responsive Strategy
- Semantic Styling
- Component Styling
- Dark Mode Strategy

Mention implementation only when proven, and classify it as replaceable, optional, framework-specific, or implementation-specific as appropriate.

---

# Extension Points

Every template must explain how to safely extend:

- Feature
- Module
- Domain
- API
- Store
- Service
- Validation
- Testing
- Configuration
- Integration
- UI Component
- Design System

Focus on engineering rules, not implementation tutorials.

---

# Anti Pattern Discovery

Extract reusable anti-patterns when supported by evidence or established engineering principles reflected by the memory set.

Examples:

- View -> API
- Store -> UI
- Business Logic -> Component
- Feature Utility -> Shared Utility
- Circular Dependency
- Large Components
- Large Services
- Shared Mutable State
- Hardcoded Configuration

Anti-patterns must remain reusable engineering guidance, not project gossip.

---

# Evolution Rules

Templates should explain how the architecture evolves safely.

Examples:

- split routing when features increase
- split state management when domains expand
- split transport layer when APIs grow
- extract shared libraries when reuse increases
- extract modules when bounded contexts become independent

Evolution guidance must remain reusable and architecture-first.

---

# Technology Classification

Every discovered technology should be classified, when supported by evidence, as one of:

- Architecture
- Convention
- Implementation
- Replaceable Dependency
- Framework Dependency
- Optional Dependency
- Project-specific Dependency
- Business-specific Dependency

Never confuse implementation with architecture.

# Required Template Files

## TEMPLATE_PROFILE.md

This projection owns:

- template name
- template purpose
- source project memory
- supported project type
- recommended tech stack
- package manager
- runtime requirements
- intended use cases
- unsupported use cases
- architecture summary
- reusable engineering context

## TEMPLATE_WORKFLOW.md

This is the most detailed projection.

It must explain reusable flows such as:

- project generation flow
- folder creation flow
- file creation flow
- routing flow
- API flow
- database/model flow
- validation flow
- testing flow
- how to add a new feature
- how to add a new route
- how to add a new API client or endpoint
- how to add a new module/component
- how to extend the template safely
- how architecture should evolve safely

## FOLDER_STRUCTURE.md

This projection owns recommended folder layout and workspace structure:

- root-level folders
- source folders
- feature/module folders
- shared/common folders
- config folders
- test folders
- generated folders
- ignored folders

## FILE_STRUCTURE.md

This projection owns reusable file and naming conventions:

- required files
- optional files
- file naming rules
- feature file patterns
- config file patterns
- test file patterns

## CODE_STRUCTURE.md

This projection owns code organization rules:

- layering convention
- component/module structure
- business logic placement
- validation placement
- type/interface placement
- error handling pattern
- reusable utility pattern
- engineering contracts
- anti-pattern boundaries

## ROUTING_STRUCTURE.md

This projection owns routing conventions:

- route file location
- route naming
- path naming
- lazy loading convention
- nested routing convention
- route guard convention if proven
- menu/navigation relationship if proven
- routing evolution guidance

## API_STRUCTURE.md

This projection owns API conventions:

- API client or endpoint structure
- request/response model pattern
- DTO/schema convention
- error response handling
- authentication header pattern if proven
- environment base URL convention if proven
- transport and integration boundaries

## DATABASE_STRUCTURE.md

This projection owns database conventions when applicable:

- entity/model pattern
- schema location
- migration convention
- repository/data-access pattern
- naming convention
- relationship convention

If the source project has no database layer, mark this as `Not Applicable`.

## TESTING_STRUCTURE.md

This projection owns testing conventions:

- test framework
- test folder layout
- unit test pattern
- integration test pattern
- e2e test pattern
- test naming convention
- test command surface
- testing evolution guidance

## GENERATION_RULES.md

This projection defines how `project-onboarding` should use the template:

- required input variables
- optional input variables
- placeholder replacement rules
- files to rename
- files to update
- commands to run after generation
- validation checks
- failure handling

This file must guide template application, not scaffold a project directly.
