# Template Generator

## Overview

Use this skill to convert stable project memory into a reusable project template blueprint.

This skill does **not** scan raw source code directly unless clarification is truly required. It consumes `.docs/` generated by `project-scanner`, treats Stable Memory as the only source of truth, and projects a reusable `.template/` from an internal engineering blueprint.

This skill is a **Template Blueprint Builder**: the generated template must be an engineering blueprint, not a generalized copy of the source project.

**Announce at start:**

> spark template 🧩 Using template-generator to build reusable project blueprint

---

# Project Memory Input

Read `.docs/` first.

If the workspace contains multiple `.docs/` directories, identify which valid project root should be templated and use that project's memory set only. Do not merge multiple project memories into one template unless the developer explicitly asks for a composed workspace template.

Expected project memory set:

```txt
.docs/
├── PROJECT_SCAN.md
├── PROJECT_PROFILE.md
├── ARCHITECTURE_GRAPH.md
├── WORKSPACE_MAP.md
├── FEATURE_MAP.md
├── API_CONTRACT.md
├── DATABASE_CATALOG.md
├── BUSINESS_FLOW.md
├── DOMAIN_MAP.md
└── TESTING_STRATEGY.md
```

Stable Memory is the only source of truth.

Do **not** scan the repository unless clarification is truly required because the memory is incomplete or contradictory.

# Output Contract

Generate exactly these files and no others:

```txt
.template/
├── TEMPLATE_PROFILE.md
├── TEMPLATE_WORKFLOW.md
├── FOLDER_STRUCTURE.md
├── FILE_STRUCTURE.md
├── CODE_STRUCTURE.md
├── ROUTING_STRUCTURE.md
├── API_STRUCTURE.md
├── DATABASE_STRUCTURE.md
├── TESTING_STRUCTURE.md
└── GENERATION_RULES.md
```

You must physically create `.template/` and write the files to disk.

Do not create new permanent files.

# Template Thinking Rules

Before generating templates, apply these rules:

- prefer engineering pattern over project structure
- prefer convention over implementation
- prefer architecture over technology
- prefer generic contract over concrete naming
- prefer reusable pattern over business logic
- prefer architecture decisions over folder names
- extract why before how
- extract reusable engineering knowledge before implementation details
- evidence overrides assumptions
- unknown is better than incorrect
- never generalize unsupported patterns

These rules are mandatory.

# Rules

- Read `.docs/` first.
- Do not generate a template if `.docs/` is missing or incomplete.
- Do not scan raw source code unless clarification is truly required.
- Do not generate implementation code.
- Do not generate a new project. That is the responsibility of `project-onboarding`.
- Do not copy project-specific business data directly into the template.
- Convert concrete names into reusable placeholders.
- Preserve proven architecture, structure, routing, API, database, testing, and UI architecture conventions.
- Mark uncertain conclusions as `Needs Review`.
- Keep output deterministic.
- Keep output polyglot and framework agnostic.

# Pattern-to-Blueprint Pipeline

```txt
Stable Memory -> Discovery -> Validation -> Generalization -> Abstraction
-> Decision -> Variability -> Blueprint (never persisted) -> Projection -> Template Files
```

A pattern advances only when the step before it supports it with evidence. Insufficient evidence at any step → mark `Needs Review`; never invent or generalize an unsupported pattern.

**Discovery** (Stable Memory only, not raw source) — architecture layers, workspace/module/folder conventions, file naming, routing, API/persistence patterns, testing strategy, validation placement, configuration strategy, integration boundaries, design-system structure, component hierarchy, extension points, anti-patterns, evolution rules. Prefer conventions that repeat across the memory set over isolated implementation details.

**Validation** — reusable only if evidenced by Stable Memory, repeated enough to count as a convention, and not architecture/framework-specific, optional, replaceable, business-specific, or implementation leakage disguised as a pattern.

**Generalization** — strip business specifics, keep the shape: `EmployeeProfile.vue` → `<FeatureName>Profile.<view_ext>`, `GET /employee` → `GET /<resource>`, `EmployeeModule` → `<FeatureName>Module`. Generalize names, not architecture — never carry over project-specific feature names, endpoints, org rules, or business workflows dressed as universal patterns.

**Architecture Abstraction** — presentation / application / state / integration / infrastructure layers, before naming any framework. Architecture is always the first layer; implementation is always the last, offered only as supporting evidence.

**Engineering Decision Extraction** — for each convention, determine why it exists, what it solves, and whether it's mandatory, optional, replaceable, framework-, or architecture-specific. When evidence is strong, express it as a contract with `MUST` / `SHOULD` / `MAY` / `MUST NOT` (e.g. "Views MUST NOT access transport clients directly," "Business logic MUST stay outside reusable UI components"). Insufficient evidence → `Needs Review`. Never invent decisions.

**Variability Analysis** — classify each pattern only when evidenced: Mandatory, Optional, Replaceable, Framework-specific, Architecture-specific, Project-specific, Business-specific (e.g. authentication, UI library, state management, transport layer, styling strategy, testing framework, build tool). Improves adaptability without collapsing architecture into implementation.

**Engineering Blueprint** — the shared reasoning layer: validated patterns, architecture abstractions, decisions, variability classifications, extension/evolution rules, anti-patterns, UI architecture patterns. Temporary — must never be written as its own file.

**Template Projection** — the ten `.template` files read from that one blueprint: no file invents its own rules, terminology stays consistent across the set, implementation appears only as replaceable supporting evidence. Increase knowledge density, not document length.

# Design System Extraction

Extract reusable design-system architecture when supported by Stable Memory — never simply copy UI implementation:

- Design System Strategy
- Component Hierarchy
- Component Ownership
- Design Tokens
- Spacing Strategy
- Typography Strategy
- Theme Strategy
- Layout Strategy
- Responsive Strategy
- Accessibility Strategy
- Interaction Pattern
- Visual Composition Pattern

Must stay business-neutral (business-specific styling never becomes reusable design-system knowledge) and technology-agnostic — reusable UI architecture, not a Vue/Tailwind/library-specific implementation.

---

# Reusable Component Extraction

Extract reusable component architecture such as Primitive Components, Composed Components, Feature Components, Page Templates, Application Views. Document ownership and responsibility for each layer when evidenced; avoid implementation-specific assumptions.

---

# Styling Strategy Abstraction

Do not assume a specific styling tool or UI library. Extract only when supported by evidence: Styling Strategy, Theme Strategy, Token Strategy, Responsive Strategy, Semantic Styling, Component Styling, Dark Mode Strategy. Mention implementation only when proven, classified as replaceable, optional, framework-specific, or implementation-specific as appropriate.

---

# Extension Points

Every template must explain how to safely extend: Feature, Module, Domain, API, Store, Service, Validation, Testing, Configuration, Integration, UI Component, Design System. Focus on engineering rules, not implementation tutorials.

---

# Anti Pattern Discovery

Extract reusable anti-patterns when supported by evidence or established engineering principles reflected by the memory set.

Examples:

- View -> API
- Store -> UI
- Business Logic -> Component
- Feature Utility -> Shared Utility
- Circular Dependency
- Large Components
- Large Services
- Shared Mutable State
- Hardcoded Configuration

Anti-patterns must remain reusable engineering guidance, not project gossip.

---

# Evolution Rules

Templates should explain how the architecture evolves safely, e.g.: split routing when features increase, split state management when domains expand, split transport layer when APIs grow, extract shared libraries when reuse increases, extract modules when bounded contexts become independent. Guidance must stay reusable and architecture-first.

---

# Technology Classification

Every discovered technology should be classified, when evidenced, as one of: Architecture, Convention, Implementation, Replaceable Dependency, Framework Dependency, Optional Dependency, Project-specific Dependency, Business-specific Dependency. Never confuse implementation with architecture.

---

# Required Template Files

Each file owns a distinct slice of the blueprint — no overlap, no reinvention:

- **TEMPLATE_PROFILE.md** — template name, purpose, source project memory, supported project type, recommended tech stack, package manager, runtime requirements, intended/unsupported use cases, architecture summary, reusable engineering context.
- **TEMPLATE_WORKFLOW.md** (most detailed projection) — project generation flow, folder/file creation flow, routing flow, API flow, database/model flow, validation flow, testing flow, and how to: add a feature, add a route, add an API client/endpoint, add a module/component, extend the template safely, evolve the architecture safely.
- **FOLDER_STRUCTURE.md** — root-level, source, feature/module, shared/common, config, test, generated, and ignored folders.
- **FILE_STRUCTURE.md** — required files, optional files, file naming rules, feature/config/test file patterns.
- **CODE_STRUCTURE.md** — layering convention, component/module structure, business logic placement, validation placement, type/interface placement, error handling pattern, reusable utility pattern, engineering contracts, anti-pattern boundaries.
- **ROUTING_STRUCTURE.md** — route file location, route/path naming, lazy-loading convention, nested-routing convention, route guard convention (if proven), menu/navigation relationship (if proven), routing evolution guidance.
- **API_STRUCTURE.md** — API client/endpoint structure, request/response model pattern, DTO/schema convention, error response handling, auth header pattern (if proven), environment base-URL convention (if proven), transport/integration boundaries.
- **DATABASE_STRUCTURE.md** — entity/model pattern, schema location, migration convention, repository/data-access pattern, naming convention, relationship convention. Mark `Not Applicable` if the source project has no database layer.
- **TESTING_STRUCTURE.md** — test framework, test folder layout, unit/integration/e2e test patterns, test naming convention, test command surface, testing evolution guidance.
- **GENERATION_RULES.md** — how `project-onboarding` should use the template: required/optional input variables, placeholder replacement rules, files to rename/update, post-generation commands, validation checks, failure handling. Guides template application — does not scaffold a project directly.

# Blueprint Validation

Before generating templates, verify:

- no business terminology remains unless deliberately placeholderized
- no project-specific endpoints remain
- no project-specific feature names remain
- no organization-specific conventions remain
- no implementation leakage exists
- all placeholders are reusable
- all engineering contracts are internally consistent
- all architecture remains reusable
- design system remains business-neutral
- component ownership is preserved
- no project-specific styling leaks into reusable layers

If any check fails, fix the blueprint before projection.
