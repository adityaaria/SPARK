# Project Onboarding

## Overview

Use this skill to generate a new project from an approved reusable template blueprint.

This skill does **not** invent a project structure from scratch. It must use an existing `.template/` blueprint or an explicitly approved template source.

**Announce at start:**

> spark onboarding 🚀 Using project-onboarding to generate project from template

## Responsibility

This skill is a **Project Generator**.

It converts:

```txt
Reusable Template Blueprint
↓
New Project
```

It must not create or normalize templates. That is the responsibility of `template-generator`.

## Required Input

Before generating a project, identify:

* project name
* target directory
* template source
* project type
* package manager
* runtime requirements
* environment name if applicable

If required input is missing, ask only for the missing critical information.

## Accepted Template Sources

Template source may be:

```txt
.template/
templates/<template-name>/
approved baseline repository
approved local template directory
```

The template must include at minimum:

```txt
TEMPLATE_PROFILE.md
TEMPLATE_WORKFLOW.md
GENERATION_RULES.md
```

If these are missing, stop and ask the developer to run `template-generator` first.

## Generation Flow

```txt
Developer Request
↓
Resolve Template
↓
Read Generation Rules
↓
Validate Target Directory
↓
Create Project Folder
↓
Copy Template Skeleton
↓
Apply Placeholders
↓
Rename Files/Folders
↓
Update Metadata
↓
Install Dependencies if instructed
↓
Run Validation Commands
↓
Run project-scanner
↓
Hand Off Result
```

## Safety Rules

Never:

* overwrite an existing project without explicit confirmation
* delete existing files without explicit confirmation
* generate a project without an approved template
* mix multiple templates unless explicitly instructed
* install unverified dependencies
* silently ignore failed validation commands

Always:

* preserve template conventions
* follow `GENERATION_RULES.md`
* report changed files
* report skipped steps
* report failed commands
* generate project memory after successful project creation

## Placeholder Rules

Replace placeholders consistently.

Common placeholders:

```txt
<ProjectName>
<project-name>
<project_name>
<FeatureName>
<feature-name>
<feature_name>
<resource>
<resource_plural>
<PackageName>
<package-name>
```

Project naming should be applied to:

* folder name
* package name
* README title
* environment examples
* route prefixes if applicable
* module names if applicable
* app metadata if applicable

## Required Output

Generated project should contain:

```txt
<new-project>/
├── source files
├── config files
├── README.md
├── .env.example if applicable
└── .docs/
```

The `.docs/` folder must be generated by running `project-scanner` after project creation.

## Validation

Run validation commands only if they are defined by the template or detected from the generated project.

Examples:

```txt
install
lint
typecheck
test
build
format check
```

If validation cannot run, document why.

## Final Response Format

After onboarding, respond with:

```txt
Project Created:
- name
- path
- template used

Generated Files:
- summary

Validation:
- commands run
- passed/failed/skipped

Project Memory:
- .docs generated or not
- reason if not generated

Next Steps:
- developer actions
```
